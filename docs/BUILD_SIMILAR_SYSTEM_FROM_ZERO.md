# 从零到一搭建“同类问答系统”实战教程（超详细）

> 目标：教你独立做出一个和本项目同类型的系统（知识库检索 + 本地模型回答 + 管理后台）。  
> 适用对象：会基本命令行，但没有完整做过 AI 问答系统的人。

---

## 0. 你将做出什么

最终产物是一个完整可运行系统，包含：

- 学生问答页面（支持普通提问和流式输出）
- 管理后台（配置模型、上传知识库、查看预览）
- 后端检索服务（TF-IDF + 关键词）
- 本地模型调用（Ollama/OpenAI Compatible）
- 可部署到学校/公司内网的服务结构

---

## 1. 先做需求定义（不要跳过）

任何系统开工前，必须写清楚 5 件事：

1. **用户是谁**：学生、老师、咨询客服？
2. **问题范围**：只答专业问题，还是允许通用知识？
3. **拒答策略**：资料不足时怎么说，是否给人工联系方式？
4. **更新流程**：谁上传知识库？多久更新一次？
5. **验收标准**：你怎样判断“系统可上线”？

推荐最小验收标准：

- 可连续稳定回答 20 个问题不崩
- 后台上传知识库 1 分钟内生效
- 出现异常时有可读错误提示

---

## 2. 技术路线选择（给你结论）

如果你追求“快搭建 + 易维护”，推荐直接选：

- 前端：Next.js + TypeScript
- 后端：FastAPI（Python）
- 检索：scikit-learn TF-IDF + jieba 分词
- 模型：Ollama（本地模型）
- 数据：JSON 文件（后续可升级数据库）

这是本项目的路线，优点是学习成本低、部署成本低、可快速迭代。

## 3. 先搭骨架项目（Day 1）

## 3.1 创建目录结构

```text
my-qa-system/
├─ frontend/
├─ backend/
├─ scripts/
├─ docs/
├─ data-source/
└─ README.md
```

## 3.2 初始化前端

```powershell
npx create-next-app@latest frontend --typescript --eslint
```

## 3.3 初始化后端

在 `backend/` 下准备：

- `main.py`（FastAPI 入口）
- `knowledge.py`（检索）
- `llm.py`（模型调用）
- `config.py`（配置管理）
- `requirements.txt`

安装依赖示例：

```powershell
py -m pip install fastapi uvicorn openai scikit-learn jieba pandas openpyxl python-dotenv
```

## 4. 先把“知识库”打通（Day 1-2）

## 4.1 设计知识条目结构

最小结构建议：

```json
[
  {
    "id": 1,
    "question": "软件工程专业核心课程有哪些？",
    "answer": "..."
  }
]
```

## 4.2 实现基础加载逻辑

目标：

- 系统启动时加载 JSON
- 过滤空问题/空答案
- 记录总条目数

## 4.3 实现混合检索

建议用两条线并行算分：

1. TF-IDF 相似度
2. 关键词匹配分数

最后加权合并（例如 `0.7 * tfidf + 0.3 * keyword`），取 TopK 返回。  
这样比单一算法更稳。

---

## 5. 再接“本地大模型”能力（Day 2）

## 5.1 选择模型和原则

- 默认 `qwen2.5:3b`（快）
- 机器够强时可切 `qwen2.5:7b`（质量更高）

## 5.2 封装模型调用模块

`llm.py` 要做三件事：

1. 读取运行配置（base_url/model）
2. 组装 prompt（有资料优先用资料）
3. 返回普通回答 + 流式回答两个函数

重要原则：

- 资料命中时：优先依据资料
- 未命中时：可给通用建议，但提醒以官方通知为准
- 失败时：返回友好提示，不把堆栈直接给用户

---

## 6. 构建后端 API（Day 2-3）

必须实现的最小 API：

- `POST /api/chat`：一次性返回答案
- `POST /api/chat/stream`：SSE 流式返回
- `GET /api/health`：健康检查
- `POST /api/admin/login`：后台登录
- `GET/POST /api/admin/config`：读取/保存配置
- `POST /api/admin/test-connection`：测试模型连接
- `POST /api/admin/upload-knowledge`：上传知识库
- `GET /api/admin/knowledge/preview`：知识预览

后端开发顺序建议：

1. 先做 `health`
2. 再做 `chat`
3. 再做 `chat/stream`
4. 最后做 admin 系列接口

---

## 7. 构建前端页面（Day 3-4）

## 7.1 用户页（聊天）

最小功能：

- 输入框 + 发送按钮
- 消息列表（用户消息/助手消息）
- 正在思考状态（loading）
- 失败提示（网络错误、模型错误）

加分功能：

- 流式打字效果
- 一键清空对话
- 快速提问卡片

## 7.2 管理页（admin）

最小功能：

- 管理员登录
- 配置模型参数并保存
- 测试模型连接
- 上传知识库文件
- 查看知识库条目预览和搜索

不要一开始就堆复杂权限系统，先保证流程闭环。

---

## 8. 配置与脚本体系（Day 4）

## 8.1 配置分层

建议采用：

- `.env`：默认配置
- `runtime_config.json`：后台动态配置

读取优先级建议：`runtime_config.json > .env > 代码默认值`

## 8.2 启动脚本化

至少准备 5 个脚本（和本项目一致）：

- 启动 Ollama
- 拉取模型
- 切换模型配置
- 启动后端
- 启动前端

为什么要脚本化：

- 新手只要执行命令，不需要理解太多细节
- 缓存路径可统一重定向，避免污染系统盘

---

## 9. 知识库上传与预处理（Day 4-5）

## 9.1 支持格式

- `.xlsx`
- `.json`
- `.csv`

## 9.2 上传流程建议

1. 后台上传文件到后端接口
2. 后端解析并校验数据
3. 丢弃无效条目
4. 覆盖写入 `qa_knowledge.json`
5. 重新加载检索索引
6. 返回“生效条目数”

## 9.3 预处理脚本建议

保留独立脚本（例如 `backend/preprocess.py`），用于把原始 Excel 批量转换成统一 JSON，便于批处理导入。

---

## 10. 质量保障（Day 5）

最少做这 3 类验证：

1. **功能验证**
   - 聊天可用
   - 上传可用
   - 配置保存可用
2. **异常验证**
   - 模型未启动时是否有友好提示
   - 上传空文件是否正确报错
3. **性能验证**
   - 首次响应时间
   - 连续提问响应稳定性

---

## 11. 部署路径（Day 6）

## 11.1 本地演示部署

- 一台 Windows 电脑
- 4 个窗口：Ollama、后端、前端、运维操作

## 11.2 学校/机构部署

推荐结构：

- Caddy/Nginx：80/443
- Next.js：3000（仅内网）
- FastAPI：8000（仅内网）
- Ollama：11435（仅内网）

上线前检查：

- 管理密码已修改强口令
- 备份策略已启用
- 对外只暴露 HTTPS 入口

## 13. 你可以直接复用本项目哪些设计

可直接照搬的成熟思路：

- `scripts/` 脚本化启动与缓存重定向
- `backend/knowledge.py` 混合检索方案
- `backend/llm.py` 上下文优先 + 通用回答兜底 prompt 结构
- `frontend/src/app/admin/page.tsx` 的后台交互流程

---

## 14. 常见失败点（提前避坑）

1. 一开始就追求“超级架构”，结果两周还没跑通
2. 只做模型调用，不做检索，答案漂移严重
3. 不做后台上传，知识更新全靠改代码
4. 不做脚本化，交接后没人会启动
5. 不做备份，知识库一覆盖就不可恢复

建议永远记住一条：**先可用，再优雅。**
